# “云境智绘”项目优化建议（基于第一阶段实践）

---

## 一、架构稳定性优化

### 1.1 修复 `@Value` 注入问题
**现状**：`AiController` 中 AI 服务地址使用了硬编码 `private String aiServiceUrl = "http://localhost:8000/analyze";`

**问题**：硬编码导致每次切换环境（本地/服务器）都需要修改代码重新编译，不符合 Spring Boot 最佳实践。

**建议**：
- 在 `BoardApplication` 上添加 `@PropertySource` 或在 `application.yml` 中确认配置项存在。
- 使用构造器注入替代字段注入，更有利于单元测试。
- 若仍然注入失败，可添加一个 `@ConfigurationProperties` 类统一管理 AI 服务配置。

### 1.2 前端配置文件抽离
**现状**：AI 服务的请求地址写死在 HTML 的 `fetch` 中，仅依赖 Vite 代理。

**问题**：部署到生产环境时，Vite 代理不再生效，需要修改 HTML 中的请求地址。

**建议**：
- 创建 `frontend/src/config.js`，根据环境变量动态切换 API 地址。
- 在 `minimal-demo.html` 或后续 Vue/React 页面中引用此配置。

### 1.3 全局异常处理
**现状**：后端仅在 `AiController` 内部 try-catch，返回 500 错误；AI 微服务使用 FastAPI 的 `HTTPException`。

**问题**：缺少统一的错误码规范和异常拦截器，调试时错误信息不一致。

**建议**：
- 后端添加 `@ControllerAdvice` 全局异常处理器，统一返回 `{"code": 500, "message": "..."}` 格式。
- AI 微服务定义统一的 `ErrorResponse` Pydantic 模型，所有接口返回一致格式。
- 前端在 `fetch` 后统一检查 `response.ok`，对网络错误和业务错误分别处理。

## 二、核心交互优化

### 2.1 手势追踪本地化部署
**现状**：MediaPipe 通过 CDN 加载 `.wasm` 和 `.tflite` 模型文件，第一阶段因网络/隐私策略导致加载失败，被迫改用鼠标模式。

**问题**：CDN 不稳定、浏览器隐私限制导致手势功能不可用，影响核心演示。

**建议**：
- 从 [MediaPipe 官方 GitHub](https://github.com/google-ai-edge/mediapipe/releases) 下载 Hand Landmark 模型的 `.wasm` 和 `.tflite` 文件。
- 放入 `frontend/public/models/` 目录，修改 `Hands` 初始化时的 `locateFile` 指向本地路径：
  ```javascript
  const hands = new Hands({
    locateFile: (file) => `/models/${file}`
  });
  ```
- 同时，在 `onResults` 中恢复手势画线逻辑（食指与拇指捏合检测），并设置合适的帧率（降低 `modelComplexity` 为 0）。

### 2.2 书写防抖与平滑算法
**现状**：鼠标画线直接连接原始坐标点，笔迹可能出现锯齿或抖动。

**问题**：手写体验粗糙，评委在查看演示视频时可能认为书写效果不佳。

**建议**：
- 实现 **一次指数平滑**（Exponential Smoothing）或 **贝塞尔曲线插值**，让笔迹更流畅。
- 添加书写速度检测，笔画过快时自动调整线条粗细，模拟真实笔锋效果。

### 2.3 圈选视觉反馈增强
**现状**：Shift+鼠标拖动时仅显示一个蓝色矩形框，松开后矩形消失。

**问题**：用户可能不确定是否成功框选，尤其在白板上笔迹颜色与选择框颜色接近时。

**建议**：
- 框选时添加**半透明遮罩层**，框内正常显示，框外稍微变暗。
- 框选完成后保留选择框（或改为虚线框），直到用户点击发送或取消。

## 三、功能完整性优化

### 3.1 用户登录与板书历史
**现状**：第一阶段无用户系统，所有板书无持久化存储。

**问题**：无法区分不同用户的板书，演示时缺少“个人教学数据”的展示。

**建议**：
- 使用 Spring Security + JWT 实现轻量登录（可先做一个固定账号演示用）。
- 建立 `board_sessions` 表，每次打开页面自动创建一个会话，保存用户从开始书写到清屏的全部截图。
- 前端添加“历史记录”侧边栏，展示当前用户的所有板书会话，点击可恢复画布。

### 3.2 多类型 Prompt 切换
**现状**：前端发送时 `whiteboard_type` 固定为 `"general"`。

**问题**：AI 无法根据用户实际书写内容（数学公式、代码、物理图）切换最优 Prompt，导致识别效果不稳定。

**建议**：
- 在工具栏添加“内容类型”下拉选择器（数学/代码/物理/通用），用户可在发送前手动指定。
- 或在前端调用 AI 之前，先用一个轻量分类器自动判断内容类型再设置对应的 `whiteboard_type`。

### 3.3 识别结果的历史保存
**现状**：AI 返回结果仅展示在右侧面板，关闭页面后丢失。

**问题**：用户无法回顾之前的 AI 分析结果。

**建议**：
- 在 `board_snapshots` 表中存储每次发送的 Base64 截图和 AI 返回的 JSON 结果。
- 前端在“历史记录”中展示缩略图和识别摘要，点击可查看完整结果。

## 四、体验与可用性优化

### 4.1 响应式布局适配
**现状**：页面仅做了基础的两栏布局，在小屏幕（平板/手机）上右侧面板可能挤压或遮挡画布。

**问题**：演示时可能在移动设备上展示，体验差。

**建议**：
- 使用 CSS `@media` 查询，在小屏幕（<768px）时将两栏改为上下布局：上方画布，下方控制面板。
- 工具栏按钮在移动端适当增大触控区域（`min-height: 44px`）。

### 4.2 AI 处理等待状态优化
**现状**：点击发送后右侧显示“正在请求 AI...”，但无进度或估计时间。

**问题**：AI 响应可能需 3-5 秒，用户可能认为卡死。

**建议**：
- 添加**加载动画**（如旋转图标或进度条）。
- 若请求超过 10 秒，显示“AI 正在深度思考中，请稍候...”。
- 前端设置 30 秒超时，超时后提示“请求超时，请检查网络后重试”。

### 4.3 快捷键支持
**现状**：所有操作均需鼠标点击。

**问题**：演示时频繁切换鼠标和键盘，操作不够流畅。

**建议**：
- `Ctrl+Z` 撤销上一步画线。
- `Ctrl+S` 直接发送当前框选区域（或发送整个画布）。
- `Escape` 取消当前框选。

## 五、赛事准备优化

### 5.1 演示视频录制脚本优化
**现状**：第一阶段仅有一个粗糙的可交互页面。

**问题**：4C 大赛要求 10 分钟演示视频，需要展示“痛点 → 亮点 → 技术原理 → 总结”。

**建议**：
- 录制前确保手势追踪可用（本地部署后），展示**隔空写字**的核心卖点。
- 设计一个完整的教学场景：老师在白板上写数学公式 → 圈选 → AI 给出推导步骤 → 老师点评修改。
- 视频中穿插架构图的简要讲解（三服务 + 智谱 API），满足“技术原理解析”的采分点。

### 5.2 作品设计说明书填充
**现状**：第一阶段跑通了基础链路，但文档只有框架。

**问题**：4C 大赛的作品设计说明书是重要评分项，需尽早开始填充。

**建议**：
- 将第一阶段的实际架构（AI 微服务 + 后端中继 + 前端）写入“系统设计”章节。
- 把纠错文档中的 **E1-E9** 问题与解决过程整理为“开发中的技术难点与创新点”写入报告，体现工程实践能力。
- 在“技术路线”中写明选型理由（为什么用 FastAPI 而不是 Flask、为什么用 MediaPipe 而不是 OpenCV）。

### 5.3 部署到云服务器
**现状**：三个服务仅在本机 `localhost` 运行。

**问题**：大赛要求提供可访问的网址，评委需要在外网体验。

**建议**：
- 将 AI 微服务和后端部署到同一台云服务器（如阿里云 ECS），分别用 `systemd` 或 `screen` 保持后台运行。
- 前端构建后部署到 Netlify / Vercel 或同一服务器的 Nginx 目录下。
- 配置 Nginx 反向代理，将 `/api/` 请求转发到后端 8080 端口，`/ai/` 请求转发到 AI 微服务 8000 端口。

## 优先级排序建议

| 优先级       | 优化项                    | 理由                           |
| ------------ | ------------------------- | ------------------------------ |
| P0（立即）   | 手势追踪本地化部署        | 核心交互亮点，直接影响演示效果 |
| P0（立即）   | 部署到云服务器            | 大赛提交截止前必须完成         |
| P1（本阶段） | 用户登录与板书历史        | 形成完整的产品体验，满足评分项 |
| P1（本阶段） | AI 处理等待状态优化       | 避免评委体验时误认为系统卡死   |
| P2（后续）   | 书写防抖与平滑            | 提升交互质感，但非阻塞         |
| P2（后续）   | 全局异常处理              | 提升系统健壮性                 |
| P3（赛前）   | 演示视频录制 + 说明书填充 | 直接影响比赛成绩               |

---

以上优化建议覆盖了从技术深度、用户体验到比赛策略的完整维度，可根据团队人力和时间灵活排期。第一阶段打下的“正方体”骨架为此提供了坚实的迭代基础。
